## R1 — Fix slow startup

### What Shalom said
> "fix loading time way too slow"

### Diagnosis
Two layers contribute to the slow feel:

1. **Main process blocking:** `main.js:298-310` eagerly calls `getSystemInfo()` during `app.whenReady()`. The PowerShell WMI script takes 2-5s. The window is created but stays empty until system info finishes (audit finding **A5**).
2. **Renderer has no loading skeleton:** While main is collecting system info, the renderer just shows a static "Loading..." text — no spinner, no skeleton, no progress signal.

### Fix
- **Main side:** Move `getSystemInfo()` call out of `whenReady()`. Trigger it via IPC AFTER the window's `did-finish-load` event fires. Window appears immediately, then collects in the background.
- **Renderer side:** Add a proper loading state in the system-info section — skeleton placeholder + spinner + "Collecting system info..." text. When IPC returns, fade in the real data.

### Files affected
- `src/main/main.js` (or after Phase 2: `src/main/index.js` + `src/main/ipc/system-info.js`)
- `src/renderer/index.html` (or after Phase 2: `src/renderer/scripts/modules/system-info-controller.js` + `styles/components.css`)

### Definition of done
- Window appears within ~500ms of clicking the icon (currently ~3-5s)
- Loading skeleton visible while system info collects
- Smooth transition when data lands

### Risk
Low. Change is mostly removing a blocking call. Loading skeleton is additive.

### Effort
Small (S)

## R2 — In-app ticket confirmation (replace Windows native dialog)

### What Shalom said
> "the confirmation for a ticket should be self-contained not the windows default system one, it looks like a error"

### Diagnosis
After ticket submission, the app calls `dialog.showMessageBox` from main process. That renders a Windows-native dialog that:
- Has a generic Windows look (often interpreted as an error or system warning)
- Doesn't match the app's branding
- Blocks until user clicks OK
- Audit finding **C13:** The dialog can also LIE — `mode: 'no-cors'` makes the fetch always look successful even if Zoho rejected it

### Fix
- Replace `dialog.showMessageBox` with an in-app **modal confirmation** styled to match the app brand
- Modal shows: success icon (green check), "Your ticket has been submitted", "We'll be in touch shortly", and a primary "OK" button
- For errors: same modal pattern with red icon and error message
- **Honesty fix (C13):** Change copy from "Ticket Submitted Successfully" (which we can't actually verify due to no-cors) to "Submission sent — you'll get an email confirmation from Zoho Desk shortly". Sets correct expectation.

### Open question for Shalom
- **Modal or toast?** Modal blocks the user (must click to dismiss) and feels more formal. Toast slides in from a corner and auto-dismisses (less intrusive). My pick: **modal for confirmation** (the user wants to know it worked), **toast for screenshot notifications** (transient).

### Files affected
- After Phase 2: `src/renderer/scripts/modules/dialog.js` (new, in-app modal helper) + `modules/ticket-form.js` (call new modal instead of IPC dialog) + `styles/modal.css`
- `src/main/main.js` — remove the `show-message-box` IPC handler if no longer needed

### Definition of done
- After submitting a ticket, no Windows native dialog appears
- An in-app branded modal shows success/error
- Copy reflects honest state ("submission sent", not "ticket created")
- Visual matches the existing app design language (purple/gradient header)

### Risk
Low. Self-contained UI change.

### Effort
Small-Medium (S-M)

## R3 — Auto-attach screenshot to ticket form

### What Shalom said
> "the screenshot should automatically be added into the form once created"

### Current behavior
- User clicks "Take Screenshot"
- Screenshot saves to `%TEMP%\SolveIT-Screenshots\`
- A toast/notification confirms it was saved
- User has to **manually drag/upload** the screenshot file into the Zoho ticket form's attachment area (and we already know drag-drop is RDP-flaky)

### New behavior
- User clicks "Take Screenshot"
- Screenshot saves AS NOW
- File is **automatically added** to the Zoho form's attachment list
- Inline toast says "Screenshot attached to ticket"
- User can remove the attachment if they don't want it

### Implementation
The Zoho WebToCase form uses standard HTML `<input type="file">` elements (`zsattachment_1` through `zsattachment_5`). We can programmatically:
1. Read the screenshot file from disk via main process IPC
2. Build a `File` object in the renderer using the file's bytes
3. Use `DataTransfer` to attach to the next available file input
4. Fire a `change` event so Zoho's JS picks it up and updates the visual list

This bypasses the drag-drop limitation entirely (works even over RDP).

### Edge cases
- **Form already has 5 attachments:** Show toast "Max 5 attachments — remove one to add screenshot"
- **Screenshot fails to capture:** Existing error path stays (no attach attempted)
- **Multiple screenshots in one session:** Each one auto-attaches to the next slot

### Files affected
- After Phase 2: `src/renderer/scripts/modules/ticket-form.js` (new `attachScreenshot(filePath)` method) + `modules/screenshot-controller.js` (call attach after capture) + main process screenshot IPC needs to optionally return the file bytes (not just the path) OR a new IPC `read-file-as-base64`
- `src/main/main.js` — extend screenshot IPC or add file-read IPC

### Definition of done
- Take a screenshot → it appears in the Zoho form's attachment list automatically
- "Submit ticket" includes the screenshot in the upload to Zoho
- Removing a screenshot from the form removes only that one (others unaffected)
- Screenshot still saved to disk too (existing behavior, not removed)

### Risk
Medium. Touching Zoho's form internals — they expect specific event sequences. Need to test attachment list updates correctly and that submission actually includes the file.

### Effort
Medium (M)
